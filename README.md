# Refleksi Tutorial Module 2
1. Dalam latihan ini saya memperbaiki 2 code quallity issue yang dideteksi oleh sonarcloud
- Mengelompokkan dependensi berdasarkan tujuannya : pada file `build.gradle.kts`, dependensi sebelumnya tidak berurutan. hal ini dapat pemeliharaan kode di masa depan. Saya memperbaikinya dengan cara mengatur ulang dependencies dan mengelompokan jenis konfigurasi yang identik.
- Menghindari Field Dependency Injection : Pada kelas-kelas seperti `ProductController.java` dan `ProductServiceImpl.java`, dependensi sebelumnya disuntikkan langsung ke variabel menggunakan `@Autowired`. Strategi saya adalah mengubah kode tersebut menjadi Constructor Injection. Saya menghapus anotasi `@Autowired` dari variabel, menambahkan keyword final untuk memastikan imutabilitas, dan membuat sebuah constructor untuk menyuntikkan dependensi yang diperlukan.

2. Ya, saya yakin implementasi saat ini telah sepenuhnya memenuhi definisi Continuous Integration (CI) dan Continuous Deployment (CD). Untuk Continuous Integration, setiap kali ada kode baru yang dipush atau pull request dibuka, GitHub Actions secara otomatis menjalankan proses yang mengompilasi aplikasi Java, menjalankan rangkaian pengujian otomatis menggunakan JaCoCo, dan melakukan analisis kode statis yang ketat melalui SonarCloud. Hal ini memastikan bahwa hanya kode yang memenuhi standar kualitas. Lalu untuk aspek Continuous Deployment, pipeline telah dikonfigurasi sehingga setiap kode yang berhasil dimerge ke branch main akan secara otomatis memanggil webhook ke Render.

# Refleksi Tutorial Module 3
1. Dalam latihan ini saya menerapkan kelima prinsip SOLID untuk memperbaiki struktur kode dan meningkatkan kualitasnya. Pertama, Single Responsibility Principle (SRP) diterapkan dengan memisahkan controller sehingga `ProductController` hanya mengurus request tentang produk dan `CarController` khusus menangani mobil. Kedua, Open/Closed Principle (OCP) dengan memastikan controller bergantung pada interface yang memungkinkan penambahan fungsionalitas servis baru tanpa perlu memodifikasi kode controller yang ada. Ketiga, Liskov Substitution Principle (LSP) diimplementasikan dengan menghapus inheritance pada `CarController` yang awalnya mengextends `ProductController`. Keempat, Interface Segregation Principle (ISP) dijaga dengan memastikan interface CarService dan ProductService hanya berisi operasi CRUD dasar yang dibutuhkan. Terakhir, Dependency Inversion Principle (DIP) diterapkan dengan mengubah kelas-kelas seperti controller untuk bergantung pada abstraksi atau inteface melalui penggunaan constructor injection.
2. Penerapan prinsip SOLID memberikan keuntungan dalam konsistensi sistem. Dengan SRP kode menjadi jauh lebih mudah dipelihara dan dinavigasi contohnya, ketika terjadi bug pada halaman /car/list, saya dapat langsung membuka file CarController yang terpisah tanpa risiko mengganggu logika dari produk. Fleksibilitas sistem meningkat tajam berkat gabungan DIP dan OCP, karena controller yang bergantung pada interface memudahkan penggantian logika kerja. Selain itu, LSP memastikan perilaku aplikasi yang dapat diprediksi, terbukti ketika penghapusan extends mencegah /car/create untuk secara tidak sengaja memicu fungsi pembuatan produk.
3. Mengabaikan prinsip SOLID akan menghasilkan kode yang rentan terhadap bug dan sulit untuk diuji atau dikembangkan. Pelanggaran terhadap LSP terlihat jelas dalam proyek awal saat CarController yang mengextend ProductController yang menyebabkan Spring Boot memetakan rute produk (seperti /list) di bawah prefix /car, membuat aplikasi kebingungan saat pengguna mencoba mengakses daftar mobil. Tanpa DIP dan OCP, ketika controller bergantung langsung pada implementasi konkret seperti CarServiceImpl, kode menjadi sangat terikat dan perubahan kecil pada struktur penyimpanan data akan memaksa perombakan kode di berbagai tempat. Selain itu, mengabaikan SRP dengan menggabungkan banyak controller ke dalam satu file Java yang besar menyulitkan pembacaan kode.
